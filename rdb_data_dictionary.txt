1 Introduction
1.1 Purpose
This document provides an overview of the National Electronic Disease Surveillance System (NEDSS) Base
System (NBS) Reporting Database (RDB), and describes the tables that comprise it, as well as the relationships
between them. It also includes:
 An explanation of key terminology used in the NBS RDB (and throughout this document)
 An overview of the NBS RDB schema and design concepts
 Descriptions of the classes and relationships found in the RDB
 An indication of the primary and foreign keys that support the database design.
NOTE: The RDB is expected to undergo continual refinement. It is a living model for database design that will be
revised as public health information needs change, as our understanding of those needs is improved, and as
available technologies increase the applicability of automation. This document represents a ‘point in time’
snapshot of the NBS Reporting Database as of the 6.0 Release of the NBS.
1.2 Scope
This document provides a high-level overview of the NBS RDB and related processes, and describes the NBS RDB
components (tables/views), their purpose, and their behavior. The intended audience for this document is any
user of the NBS that would like to better understand the NBS RDB. Additional NBS RDB documentation that
provides more detail about the RDB tables, the ETL process, and the reporting module can be found on NBS

1.3 Key Terminology and Acronyms
Key terminology and acronyms that are used in this document are described in the table:
Term Acronym Definition
Bridge Table Bridge tables are used in a star schema in a reporting database to resolve
many-to-many relationships between fact tables.
Conformed Dimension A conformed dimension is a type of dimension table containing
information common across multiple fact tables.
Data Warehouse A data warehouse is a central repository of integrated data used for
reporting and data analysis.
Dimension Table A dimension table is a table in a star schema of a data warehouse. A
dimension table stores attributes, or dimensions, that describe the
objects in a fact table.
Extract, Transform, Load ETL The ETL process moves data from the NBS ODS database tables into the
NBS RDB tables. It extracts the highly relational operational data that are
stored in the NBS ODS; transforms the data by translating coded values
and applying other logic, as required; and loads the data into a more
flattened table structure in the RDB.
Fact Tables Fact tables consist of the measurements, metrics, or facts of a business
process. Facts are located at the center of a star schema or a snowflake
schema surrounded by dimension tables.
Lookup Tables Lookup tables in the NBS are data structures that allow fast access to
dimensional data by providing an array that replaces runtime
computation with a simpler array indexing operation.
NEDSS Base System NBS The NBS provides reporting jurisdictions with a NEDSS-compatible
information system to facilitate transferring health, laboratory, and
clinical data efficiently and securely over the Internet. More information
is available on http://wwwn.cdc.gov/nndss/nedss.html.
National Electronic Disease
Surveillance System Base System
NEDSS The NEDSS facilitates electronically transferring public health surveillance
data from the healthcare system to public health departments. It is a
conduit for exchanging information that supports NNDSS. When states
and territories voluntarily submit notifiable disease surveillance data
electronically to CDC, they use data standards and electronic disease
information systems and resources supported in part by NEDSS. This
ensures that state data shared with CDC are submitted quickly, securely,
and in an understandable form.
National Notifiable Diseases
Surveillance System
NNDSS The National Notifiable Diseases Surveillance System (NNDSS) is a
nationwide collaboration that enables all levels of public health—local,
state, territorial, federal, and international—to share notifiable disease-
related health information. Public health uses this information to
monitor, control, and prevent the occurrence and spread of state-
reportable and nationally notifiable infectious and noninfectious diseases
and conditions.
Online Analytical Processing OLAP OLAP is a class of information systems that performs multidimensional
analysis of business data and provides the capability for complex
calculations, trend analysis, and sophisticated data modeling. OLAP is
designed to provide support for data analysis via a flattened structure.
Online Transactional Processing OLTP OLTP is a class of information systems that facilitates and manages

3 RDB Schema Design
The NBS RDB is designed as a star schema. As previously mentioned, this design is meant to facilitate data
analysis vs. transactional data. A star schema provides the easy access to data and supports multidimensional
analysis. Also, a star schema design is easy to change and scale because the granularity is at the lowest level.
A star schema is characterized by one centralized fact table with data dimension points radiating from it. Each of
the points of the star is a much smaller dimension (or lookup) table. Each dimension table contains information
about the entries for a particular attribute in the fact table and is joined to the fact table using a primary
(surrogate) key to foreign key join. Dimension tables are not joined to each other. A typical fact table contains
foreign keys from dimensions. A star query is a join between a fact table and a number of dimension tables.
Designing a star schema database aids in the creation of data marts for easy data analysis.
The primary components of this schema are fact tables and dimension tables, which are further explained in the
next section.
3.1 Facts
Fact tables are central tables in the star schema (as indicated in the graphic above). In the NBS RDB, the fact
tables are the NBS events (e.g. investigation, lab report, morbidity report, etc.) and entities (e.g. patient,
provider, organization, etc.). The investigations are divided into fact tables by type of investigation and condition
or groups of conditions. Fact tables can contain a single or a composite primary key and are joined to dimension
tables through foreign keys. When the primary key is a composite key, the combination of the keys is unique for
each record.

3.2 Dimensions
Dimension tables are the points of the star schema. Each dimension table may also contain a single primary key
or a composite primary key. It may or may not contain foreign keys to other dimension tables. Dimensions are
joined to a related fact table using a primary key to foreign key join. Every dimension table has a single-field
primary key; this (surrogate) key is just an automatically incrementing number. There are four main types of
dimensions in use in the NBS RDB.
3.2.1 Primary Dimensions
In the NBS RDB, primary dimensions contain flattened data for all of the key (high-level) NBS entities and events
(e.g. D_PATIENT, D_INTERVIEW, etc.).
3.2.2 Snowflake Dimensions
The NBS RDB uses snow-flaking—that is separating information that is stored in dimension tables into
hierarchies. A snowflake schema is similar to the star schema. It normalizes dimension tables to save data
storage space and can be used to represent hierarchies of information. Many of these dimensions are multi-
value dimensions structured to comply with relational database rules, with bridge tables to resolve many-to-
many relationships to the fact tables.
These tables typically store information for questions that allow the user to select multiple responses, (e.g.
patient race), or for groups of questions that the user can respond to as a group multiple times, (e.g. patient
identification numbers). In NBS terminology, these represent repeating questions or repeating blocks of data.
3.2.3 Conformed Dimensions
Another type of dimension table contains information common across multiple fact tables, such as information
about organizations that are participating in an event. For example, a lab report captures information about the
reporting facility as well as the ordering facility; both are organizations and their information is stored in the
organization dimension tables. These dimension tables that are common across fact tables are called conformed
dimensions. Conformed dimensions are the ‘highways’ that connect fact tables.
3.2.4 Page Builder Dimensions
Page Builder dimensions contain flattened data for all of the subgroup categories of questions used within page
builder pages (e.g. D_INV_ADMINISTRATIVE, D_INV_CLINICAL, etc.).

3.3 Data Marts
Subsets of the data in the RDB have been created as radically denormalized tables, known as data marts. These
data marts exist as either physical tables or as views. Physical tables are used when complex data manipulation
is necessary to obtain the desired subset, whereas views are used when less manipulation is required, since they
require less storage space (only the query is stored and not the table; the table is generated each time the user
runs the corresponding report).
Data marts are created from the data contained in the fact and dimension tables in the RDB.
3.4 Primary and Foreign Keys
In general, the primary and foreign keys, indicated by the suffix ‘key’ in a column name, are pointers to a record
in another table, whether fact to dimension, dimension to dimension, or dimension to fact. In some instances
multiple joins may be necessary to get the desired pieces of information. The NBS RDB follows the standard
database warehousing practice of replacing the production key (key that uniquely identifies data in the
transactional database (i.e. ODS)) with a surrogate key, which is an anonymous (not known to the transactional
database), system-generated integer key value. A primary key in the RDB is a surrogate key; a composite primary
key in the RDB is a combination of surrogate keys.

4.1.1.1 Table Naming Conventions
Because the RDB has evolved over many years, naming conventions are not fully consistent across all tables and
views. However, for tables created since 2006 (and going forward), the following naming conventions apply:
 D_ starting a table name indicates a DIMENSION table.
 D_INV starting a table name indicates a PAGE BUILDER INVESTIGATION DIMENSION table.
 F_ starting a table name indicates a FACT table.
 F_S starting a table name indicates a STAGING table for FACT population.
 L_ starting a table name indicates a LOOKUP table.
 S_ starting a table name indicates a STAGING table.
 _GROUP ending a table name indicates a BRIDGE table.
 _INC ending a table name indicates an INCREMENTAL table (used for incremental ETL processing).

implemented with an SSIS package. A number of ‘interim’ tables were implemented in NBS 6.0 in support of this

4.1.2 Table Types
Tables in the NBS RDB currently fall into one of four types:
 DYNAMIC TABLE, INCREMENTALLY REFRESHED
 DYNAMIC TABLE, NOT INCREMENTALLY REFRESHED
 STATIC TABLE, INCREMENTALLY REFRESHED
 STATIC TABLE, NOT INCREMENTALLY REFRESHED
To better understand what these four table types mean, the following definitions are provided:
 A dynamic table is a table that has dynamically created columns based on metadata, (i.e. the table could
change with each ETL run, depending on changes that have been made to event pages in the Page
Builder module).
 A static table is a table that has fixed columns, which is created via scripts (vs. dynamically with each ETL
run).
 An incrementally refreshed table ONLY adds or updates records in the table if a change has occurred
since the last time the ETL process has been run for that table.
 A not incrementally refreshed table’s data are dropped and completely recreated with each ETL run.
The table below shows which tables in the NBS RDB fall into which type. Over time, the goal is for more and
more RDB tables to move to the upper left quadrant of this matrix to reduce the burden on, and thus increase
the performance of, the ETL process.
