# Comparison Fixtures (APP-471)

Synthetic `NBS_ODSE` INSERT fixtures that drive RDB_MODERN coverage through the **real** NBS
RTR pipeline, for a planned RDB-vs-RDB_MODERN comparison against MasterETL.

## What this branch does — the no-NRT-shortcut conversion

The fixtures were originally hand-authored with CDC-bypass *shortcuts* — direct `nrt_*` INSERTs
and manual `EXEC sp_*_postprocessing` calls — which made RDB_MODERN look ~90% covered. This
branch **removes every shortcut** so coverage is produced only by the genuine pipeline:

```
NBS_ODSE → SQL Server CDC → Debezium → Kafka → kafka-connect sink → nrt_*
         → reporting-pipeline-service (sp_*_event + postprocessing + datamart SPs) → RDB_MODERN
```

Fixtures are **ODSE-only**: no `nrt_*` INSERTs, no `EXEC sp_*`. Everything downstream is derived
by the service from the ODSE rows, exactly as in production.

### Headline finding
Removing the shortcut dropped measured coverage from **90.5% → 14%** — i.e. ~84 points were a
shortcut artifact, not pipeline output. That both inflated RTR coverage and confounded the
intended comparison. Faithful, fixtures-only coverage was then recovered to the **~80%** range by
fixing fixture fidelity (datamart-routing metadata, investigation↔patient links, real ODSE
chains, dedicated entities, obs-heavy lab/bmird) and by fixing genuine pipeline bugs. The full
arc and root causes are in [`NO_SHORTCUT_FINDINGS.md`](./NO_SHORTCUT_FINDINGS.md).

## RTR bugs found
Authoring against the real pipeline surfaced defects the shortcut had masked. Fixed here (TDD):

| Bug | Defect | Fix |
|---|---|---|
| **#17** | lab-test key-gen race (2627/1205 under concurrency) | `sp_getapplock` + explicit key allocation |
| **#19** | `LAB_TEST.RECORD_STATUS_CD` CHECK violation (547) | normalize the status fallback |
| **#26** | notification key-gen race (2627) | `sp_getapplock` + snapshot refresh |
| **#18** | followup-observation NPE (null `obs_domain_cd_st_1`) | null-safe comparison (cosmetic) |

Filed for follow-up (documented, not fixed here): **#16** covid_lab LOINC seed gap, **#21**
summary-case service drain race, **#22** LDF seed/chain gating, **#24** bmird multi-value
`PARTITION` cap, **#25** dyn-datamart `date`/`float` legacy-shell clash. See [`../../bugs/`](../../bugs).

> **#20 (obs-batch fail-fast)** was investigated, "fixed", then **reverted**: it is *intentional*
> defer-and-retry design, not a bug. The coverage flakiness it appeared to cause was the real
> key-gen races (#17/#26) and the dyn-datamart defect (#25) amplified by that intentional
> fail-fast — not bad fixture data. See `bugs/20_*`.

## Coverage
Regenerated by `scripts/coverage_summary.sh` into `coverage/coverage_merged.md`; currently in the
**~80%** range (≈78–81%). The exact figure varies slightly run-to-run because the service's
intentional batch fail-fast (#20) defers any batch that hits a residual poison (#25, #21). The
realistic fixtures-only ceiling (~85–89%) is gated on the documented bugs (seed-gated #16/#22,
routine #24/#25) and out-of-bounds tables (`var_datamart`, `covid_lab*`, `aggregate_report`,
`f_var_pam`, MasterETL-only).

## Running it
```sh
cd testing-tools/synthetic-odse-fixtures
./scripts/merge_and_verify.sh    # down -v; build; up; apply ODSE fixtures tier-by-tier; drain (~15-20 min)
./scripts/coverage_summary.sh    # regenerate coverage/coverage_merged.md
```
Requires only the baseline NBS images (no external dependencies).
