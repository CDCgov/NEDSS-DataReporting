# Comparison Fixtures (APP-471)

Synthetic `NBS_ODSE` INSERT fixtures that drive RDB_MODERN coverage through the **real** NBS
RTR pipeline, for a planned RDB-vs-RDB_MODERN comparison against MasterETL.

## How these fixtures work: ODSE-only

The fixtures are **ODSE-only**: they author `NBS_ODSE` rows and let the **real** NBS RTR pipeline
derive everything downstream, exactly as in production:

```
NBS_ODSE → SQL Server CDC → Debezium → Kafka → kafka-connect sink → nrt_*
         → reporting-pipeline-service (sp_*_event + postprocessing + datamart SPs) → RDB_MODERN
```

No `nrt_*` INSERTs and no `EXEC sp_*` in the fixtures. RDB_MODERN coverage is exactly what RTR
produces from the ODSE rows. Current coverage is in the **~80%** range (see [Coverage](#coverage)).

## Coverage
Regenerated by `scripts/coverage_summary.sh` into `coverage/coverage_merged.md`; currently in the
**~80%** range (≈78–81%). The exact figure varies slightly run-to-run because the service's
intentional batch fail-fast (#20) defers any batch that hits a residual poison (#25, #21). The
realistic fixtures-only ceiling (~85–89%) is gated on the documented bugs (seed-gated #16/#22,
routine #24/#25) and out-of-bounds tables (`var_datamart`, `covid_lab*`, `aggregate_report`,
`f_var_pam`, MasterETL-only).

## Running it
```sh
cd testing-tools/synthetic-odse-fixtures
./scripts/merge_and_verify.sh     # down -v; build; up; apply fixtures tier-by-tier; drain; coverage (~15-20 min)
./scripts/apply_odse_fixtures.sh  # apply fixtures onto an already-running stack (no reset, no coverage)
./scripts/coverage_summary.sh     # regenerate coverage/coverage_merged.md
```
Requires only the baseline NBS images (no external dependencies).
