# Data lineage

The `lineage/` subtree documents how each populated RDB_MODERN column traces
back through staging to its NBS_ODSE source: the end-to-end
`ODSE column → event SP / staging → nrt_* → postprocessing SP → RDB_MODERN column`
chain. It exists in two forms:

- **`lineage/LINEAGE_NARRATIVE.md`** is the human-readable version: a
  per-cluster narrative (L1–L6 plus the gap-fill sections G1–G4) covering all
  118 in-scope tables. For each table it describes how rows flow, the gating
  predicates, the notable transforms, and what is blocked or skipped, citing the
  fixture and coverage report that prove each path. Read this for onboarding or
  design review.
- **`lineage/LINEAGE_COLUMNS.tsv`** is the machine-readable appendix: one row per
  RDB_MODERN column, tracing it back to its ODSE source(s) with the writing SP,
  the fixture that establishes the mapping, and a status flag. `LINEAGE_COLUMNS.jsonl`
  is the same data as one JSON object per line, for the schema-diff tool. Both are
  **generated and gitignored**: build them from the `lineage/columns/*.tsv` source
  slices with `python3 scripts/build_lineage_columns.py`.

## Why this isn't already in the SQL

Each SQL routine contains one hop but never the whole chain. The `sp_*_event`
SPs read `nbs_odse.dbo.*` and project JSON (the ODSE-to-staging edge). The
`sp_nrt_*_postprocessing` / `sp_d_*` / `sp_*_datamart_*` SPs read `nrt_*` staging
and write RDB_MODERN (the staging-to-RDB_MODERN edge); they never read ODSE
directly (see STRATEGY.md, "postprocessing SPs read NRT staging only"). No single
place records the full `ODSE col → … → RDB_MODERN col` path. The fixture work
reconstructed it: each of the ~38 fixtures encodes which ODSE inputs light up
which RDB_MODERN columns through which SP. This lineage is the synthesis of that
knowledge, not a fresh derivation from the SP source.

## Column appendix schema

Each row of `LINEAGE_COLUMNS.tsv` traces one RDB_MODERN column:

```
rdb_modern_table | rdb_modern_col | writing_sp | nrt_staging_source | odse_source_col(s) | transform_note | status | fixture_proof
```

- **nrt_staging_source**: the `nrt_*` column (or `#tmp` derivation) the SP reads.
- **odse_source_col(s)**: the `nbs_odse.dbo.*` column(s) feeding that staging
  column, via the matching `sp_*_event` SP's JSON projection.
- **transform_note**: the CASE / COALESCE / substring / pivot / code-lookup
  applied, with an SP line cite when not obvious.
- **status**: one of `VERIFIED` (a fixture populates it and coverage confirms),
  `INFERRED` (the SP maps it but no fixture proves it), `DYNAMIC` (written via
  dynamic SQL, source not statically derivable), `MASTERETL_ONLY` (no RTR ODSE
  path, per `catalog/odse_unknown_tables.md`), or `BLOCKED:#NN` (reachable but
  capped by bug NN).
- **fixture_proof**: the fixture file that establishes the mapping, or `—` for
  INFERRED / DYNAMIC.

Every column gets a row and an honest status; an INFERRED flag is preferable to
a guessed ODSE source.

## Source artifacts

All paths relative to `testing-tools/synthetic-odse-fixtures/`.

| Artifact | What it provides |
| --- | --- |
| `catalog/rtr_target_columns.md` | Every (table, column) RTR writes plus the writing SP and Dynamic/Guarded flags. The spine of the column appendix. |
| `STRATEGY.md` | The comparison pipeline, the tier model, and the "postprocessing SPs read NRT staging only, event SPs read ODSE" convention. |
| `fixtures/10_subjects/*.sql` | Tier-1 per-subject ODSE inputs and the staging rows they drive. |
| `fixtures/20_links/*.sql` | Tier-2 cross-subject edges (act_relationship / participation / nbs_act_entity). |
| `fixtures/30_sp_coverage/*.sql` | Tier-3 datamart enrichments. Each header documents the SP, PHC/UID, and target columns it covers. |
| `coverage/coverage_merged.md` | Rolled-up coverage report (regenerated by `scripts/coverage_summary.sh`): which columns are actually populated. |
| `catalog/odse_unknown_tables.md` | Table classification (MasterETL-only / datamart-SP-driven / Tier-1-2-reachable). |
| `BLOCKED:#NN` flag | Where a column is reachable in principle but capped by an RTR bug. |
| `liquibase-service/.../005-rdb_modern/routines/*.sql` | Ground truth for a transform when a fixture or coverage report is ambiguous. |

## Regenerating the appendix

`scripts/build_lineage_columns.py` rebuilds the appendix from the per-cluster
slices in `lineage/columns/` (concatenating L1–L6 then G1–G4, deriving the
`mapping_kind` column, and emitting both `LINEAGE_COLUMNS.tsv` and
`LINEAGE_COLUMNS.jsonl`). To extend the lineage, edit or add a slice and rerun
the script.
