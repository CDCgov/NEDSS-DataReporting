# Bug #17 — nrt_lab_test_result_group_key / nrt_lab_test_key non-atomic IDENTITY key-gen race (Error 2627)

**Symptom:** intermittent `Error 2627: Violation of PRIMARY KEY constraint 'pk_nrt_lab_test_result_group_key' ... duplicate key value (1)` (and the nrt_lab_test_key analogue) thrown by `sp_d_labtest_result_postprocessing` (routine 017, ~lines 877-889) / `sp_d_lab_test_postprocessing` (018, ~880-889) at step "GENERATING keys for new TEST_RESULT_GROUP". Recorded in `RDB_MODERN.dbo.job_flow_log` (Status_Type='ERROR').

**Root cause:** the key-gen uses a non-atomic `IDENT_CURRENT(...)` + `DBCC CHECKIDENT(...,RESEED,...)` + INSERT pattern inside `BEGIN TRANSACTION`. Under concurrent / retried `PostProcessingService.processObservation` invocations this races and two callers compute the same next key → PK collision on (1). Single-threaded standalone execution runs clean — which is exactly why it is INTERMITTENT.

**Impact (large, via fail-fast):** the throw → `DataProcessingException` → `PostProcessingService.processIdCache` sets `processingFailed=true` → all lower-priority entities in that CDC batch are SKIPPED (priority OBSERVATION=14 < CONTACT=15 < TREATMENT=16 < VACCINATION=17). So `d_var_pam` (varicella), `f_vaccination`, `covid_contact_datamart`, etc. intermittently drop to ~0 depending on batch timing (the "flakiness": d_var_pam bounces 127<->26). Also blocks obs-heavy comparison fixtures (bmird antimicrobial, lab100/101) from landing reliably.

**Fix (upstream, out of bounds for fixtures):** make the key allocation atomic/idempotent (e.g. a sequence, or an `INSERT ... SELECT ISNULL(MAX)+ROW_NUMBER` with proper locking / a serializable upsert) in routines 017 & 018. Until then the race is latent for any batch that pushes enough rows through the lab key tables.

**ODSE-data mitigation (in-bounds, applied):** `morbidity.sql` set its 16 v2 component obs (20080100-20080115) to `obs_domain_cd_st_1='Result'` + `ctrl_cd_display_form=NULL`, which `PostProcessingService.extractValFromMessage` (lines 374-384) classifies as LAB_REPORT → they were fed into the lab key tables (16 spurious LAB_TEST rows + the contended group-key rows). Setting their `ctrl_cd_display_form='MorbReport'` removes them from the lab path (the dominant batch contributor), greatly reducing the race trigger frequency. Does NOT fix the underlying routine race.
